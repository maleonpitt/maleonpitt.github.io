Anatomy Content Indexing & Search Pipeline (Serverless)

Live on AWS (Terraform-managed dev environment): serverless ingestion for anatomy course files—upload → validate → chunk → embed → vector index → metadata—powering downstream RAG chat assistants. Includes a React admin UI for uploads, job history, and index management. Repo: anatomy-indexer-infra (IaC) + anatomy-indexer-app (UI).

End-to-end flow

  • Instructor signs in (Cognito) → API returns pre-signed S3 URL → browser uploads directly to the content bucket.
  • S3 event invokes a trigger Lambda → Step Functions runs: validate → detect type → parse/chunk → embed (OpenAI) → upsert (Pinecone) → write metadata (DynamoDB) → notify.
  • Admin UI polls job status and supports lifecycle actions (e.g., delete) via REST API behind API Gateway.

AWS services (by role)

  • Ingestion & orchestration — S3 (KMS-encrypted content + event notifications on PDF/Office/text uploads), Lambda (VPC-attached pipeline functions), Step Functions (retries/catches per stage), SQS (dead-letter queue for failed runs).
  • API & auth — API Gateway (REST, CORS, custom Lambda authorizer), Cognito user pool, DynamoDB (job/document lifecycle table, on-demand).
  • Secrets & encryption — Secrets Manager (OpenAI + Pinecone keys), KMS (separate keys for S3 and DynamoDB).
  • Network — VPC (private subnets, NAT, flow logs, interface endpoints for S3, DynamoDB, Secrets Manager, KMS).
  • Admin UI delivery — S3 (frontend build), CloudFront, Route 53 alias, ACM (lab hostname omitted here).
  • Observability & security — CloudWatch Logs, SNS alerts, CloudTrail, GuardDuty, Security Hub, Macie (Terraform modules).
  • IaC — modular Terraform (us-east-1); dedicated S3 buckets for content, frontend, CloudTrail logs, and remote state (names omitted).

What it enabled for the product

  • Large course files never block on API Gateway—uploads scale via S3 and async processing.
  • Repeatable, observable pipeline with per-step failure handling and instructor-facing job status in the UI.
  • Feeds the same Pinecone/OpenAI retrieval stack used by anatomy Q&A apps after indexing completes.

Also used: Python, React (Vite), Pinecone, OpenAI API

Deploy note: Infrastructure applied via Terraform (no GitHub Actions workflow in repo). Lab domains, API URLs, and bucket names are omitted.

Anatomy Course Materials Chat Assistant

Live on AWS: student Q&A over anatomy course PDFs—retrieve relevant excerpts, then generate answers grounded in uploaded materials. Single GitHub repo with anatomy-chatbot-frontend (React) and anatomy-chatbot-backend (Flask RAG API).

AWS & infrastructure (production)

  • Amazon EC2 + Docker — Flask API runs containerized on EC2 behind the load balancer; instance profile / Secrets Manager supply credentials at runtime (not baked into images).
  • Application Load Balancer (ALB) — HTTPS termination and health-checked routing to the API tier; dedicated API hostname with ACM in production.
  • Amazon S3 (multiple buckets) — frontend: private bucket for the built React SPA (CloudFront origin access control, edge caching, SPA routing to index.html); content: course-material uploads; deploy artifacts: CodeDeploy release bundles (same pattern in more than one region); Terraform state: versioned remote state. Campus IT DNS points the institutional app hostname at CloudFront; ACM provides HTTPS on that alias.
  • CloudFront /api/* behavior — same-origin API from the browser: CloudFront forwards /api to the ALB origin over HTTPS so the SPA and API share one public hostname.
  • Amazon DynamoDB (on-demand) — three application tables: user accounts (email partition key), chat sessions (session id), and interaction history (email + timestamp sort key for audit and analytics).
  • AWS Secrets Manager — JSON secret loaded into the Flask process on startup (OpenAI, Pinecone, and other keys).
  • AWS CodeDeploy — pulls backend release ZIPs from the artifacts bucket; rolls out to EC2 via appspec.yml lifecycle hooks (stop → install → start → validate).
  • Terraform — IaC for S3, CloudFront, optional custom domain/ACM, and /api/* origin to ALB; remote state in S3 with a DynamoDB lock table for safe concurrent applies.
  • ACM — TLS certificates for CloudFront (us-east-1) and/or API hostname as configured in Terraform vars.

CI/CD (GitHub Actions)

  • CI — compiles backend (py_compile) and builds the React frontend on every push/PR.
  • Terraform workflow — fmt, validate, plan; optional apply via dispatch when remote state is configured.
  • Deploy frontend (S3 + CloudFront) — npm run build, aws s3 sync, CloudFront invalidation when bucket/distribution variables are set.
  • Deploy backend (CodeDeploy) — packages backend, uploads to S3, creates CodeDeploy deployment to the EC2 deployment group.
  • AWS smoke check — manual workflow to verify DynamoDB tables and CodeDeploy app reachability.

Supporting services (non-AWS)

  • Pinecone — vector index for semantic retrieval over embedded course chunks.
  • OpenAI API — embeddings and completions for the RAG answer path.

What it enabled for the product

  • Login/signup, session-aware chat, admin PDF upload, and answers grounded in retrieved excerpts—not free-form hallucination.
  • Public app on an institutional/lab hostname (IT or Route 53 DNS → CloudFront), with edge-cached UI and API reachable on the same hostname when CloudFront routes /api to the load-balanced backend.
  • Repeatable releases: infra changes via Terraform; app changes via Actions → S3/CodeDeploy without manual SSH deploys.

Also used: React, Flask, Python, boto3

Confidentiality: Lab domains, bucket names, and account-specific identifiers are omitted.

Anatomy Grade Predictor (Web App)

Live on AWS: instructors upload assessment CSVs, train a regression model on demand, and export batch predictions with explainability charts. Monorepo with frontend (React) and backend (Flask). Instructors use a lab-hosted HTTPS URL on a private domain (not listed here).

AWS & infrastructure (production)

  • Amazon Route 53 — lab-hosted subdomain with an alias record to CloudFront (DNS shows alias → *.cloudfront.net).
  • Amazon CloudFront — HTTPS at the edge for the instructor-facing hostname; ACM validated through DNS in the same zone.
  • Amazon S3 — dedicated bucket for production UI assets (built React app) used as the CloudFront origin; separate bucket pattern in the account for predictor-related uploads or artifacts.
  • Amazon EC2 — t3.small in us-west-2 runs the Flask training/inference API (Docker); browser calls the API while static UI is served from S3 via CloudFront.
  • IAM instance role — EC2 role for AWS API access without long-lived keys in the image.
  • Docker Compose — local and server-side orchestration of frontend build workflow and backend service during development and deploy.

Application capabilities

  • Train — upload CSV → scikit-learn regression → evaluation metrics returned to the UI.
  • Predict — single-student inference or batch upload with downloadable results CSV.
  • Explainability — feature-importance chart endpoint so instructors see which inputs drive predicted grades.

What it enabled for the product

  • Rapid iteration on grading models without a separate ML platform—train and evaluate in the browser.
  • Split hosting: static UI on S3/CloudFront, ML API on EC2—typical pattern for a small departmental tool without over-building serverless.

Also used: Python, Flask, scikit-learn, pandas, React

Deploy note: Route 53, S3, and CloudFront live outside the app repo; bucket names and lab domains are omitted from this portfolio.

AI-Assisted Physical Therapy Web Platform

Web platform for therapist-led low-back rehab: enrollment, personalized plans, daily check-ins, and AI-assisted exercise discovery. Built and deployed on AWS (EC2 + ALB + VPC); stack is maintained locally in separate frontend and backend codebases—the AWS environment is no longer live.

AWS architecture (deployed; environment since decommissioned)

  • Amazon EC2 — ran Dockerized frontend and backend services via Compose on the instance for a repeatable runtime.
  • Application Load Balancer (ALB) — HTTPS entry point with health-checked routing to the app tier.
  • Amazon VPC — public/private subnets and NAT for outbound access; VPC interface endpoints supported operational access (e.g., SSM Session Manager) without public SSH.
  • Docker / Docker Compose — orchestrated the Next.js frontend (with API proxy) and separate backend API service, including a persisted volume for backend data.

Application architecture

  • Split repos — Next.js frontend and Node.js backend developed and versioned independently; Compose wired them together at deploy time.
  • Next.js API proxy — route handlers forward /api/backend/* to the backend URL so browsers stay same-origin while business logic stays on the API tier.

What it enabled for the product

  • Dual-role experience: therapist dashboard (roster, alerts, plan editing) and participant portal (ID-based access, daily check-ins, embedded video).
  • Documented, script-driven AWS deployment (manual/ops-run); no CI pipeline in the frontend repo.

Also used: Next.js, React, TypeScript, Node.js, OpenAI API, YouTube Data API

Automated Assessment Feedback & Email Delivery Portal

Instructor-facing portal (production Docker deploy on a lab Linux host): upload assessment spreadsheets, review per-student category breakdowns, generate AI narrative summaries, and send individualized HTML feedback email in bulk. Repo: React frontend + Flask API (anatomy-feedback).

End-to-end flow

  • Instructor signs in with Microsoft Entra ID (OAuth2; User.Read + Mail.Send scopes).
  • Upload category map Excel, then student results Excel → pandas groups missed items by category/module.
  • OpenAI turns structured per-student metrics into instructor-tone email copy; instructor reviews in the UI.
  • Microsoft Graph API sends HTML email to each student from the instructor’s mailbox (not a custom SMTP server).

Infrastructure & security

  • Docker Compose — backend (Flask/Gunicorn) and nginx services on one host; React build mounted read-only into Nginx.
  • Nginx — HTTPS termination (Let’s Encrypt certs on the host), static UI, reverse proxy to Flask, OAuth callback route to the API.
  • Persistent volume — session data on a host-mounted directory under the app container (not serverless storage).
  • Microsoft Entra ID — institutional login; Flask session cookies scoped for production (secure, cross-site where needed).

Application stack

  • React — instructor UI for uploads, per-student feedback cards, and send confirmation.
  • Flask + pandas + openpyxl — Excel ingest, category aggregation, batch send orchestration.
  • OpenAI API — structured summary generation from JSON feedback payloads.

What it enabled for the product

  • Turned manual post-exam feedback into a guided workflow: upload → analyze → review → one-click bulk email.
  • Used instructors’ existing Microsoft 365 identity and mail permissions—no separate email infrastructure to operate.
  • Containerized deploy suitable for a single departmental server without a full AWS serverless pipeline.

Also used: React, Flask, pandas, requests-oauthlib, Gunicorn

Deploy note: Lighter cloud footprint than the AWS-heavy anatomy projects—primary integrations are Microsoft 365 and OpenAI. Lab hostnames and credentials are omitted.

Assistive Technology Funding Program Chat Assistant

Team-built funding-program chat (believed live on Azure; public URL and full hosting layout are outside my scope). Users ask natural-language questions and receive structured program cards with eligibility-aware guidance. My work focused on the RAG backend—retrieval and grounded generation—not the Angular UI or production deploy pipeline.

My contribution — RAG on Azure

  • Azure AI Search — integrated dual indexes (programs + passages): full-text and vector retrieval, plus filters (state, insurance, age) before answers are composed.
  • Azure OpenAI — embeddings and chat completions for LLM-driven query expansion, re-ranking, and responses grounded in retrieved program and FAQ passages.
  • Dual-index search adapter — end-to-end path: expand query → search both indexes → normalize results → synthesize program-card style output (minimal hard-coded synonym rules).
  • Django REST hook — wired RAG into the chat API so the UI receives structured program recommendations backed by search results, not free-form hallucination.

Broader platform (team stack; not my primary ownership)

  • Angular frontend, OAuth2, PostgreSQL, MongoDB (sessions/messages), Azure Blob (attachments), Azure Speech (voice)—present in the repo; deployed via team processes (e.g., Azure Static Web Apps workflow for the UI).

What it enabled for the product

  • Answers tied to indexed funding-program and FAQ content, with eligibility fields surfaced for comparison across programs.
  • Managed Azure AI Search + OpenAI instead of operating a self-hosted vector database and model host.

Also used: Python, Django REST Framework

How I Work

Cloud and solutions engineering practices I apply on every project:

  • Map business requirements to AWS (or Azure) services—prefer managed, pay-for-use components where they fit.
  • Define data flows, trust boundaries, and encryption (e.g., KMS, pre-signed uploads) before wiring application code.
  • Ship infrastructure as code (Terraform) and container images so environments stay reproducible.
  • Document deployment topology (VPC, load balancers, serverless orchestration) for handoff and production support.
